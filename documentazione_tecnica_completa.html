🏗️ Architettura del Progetto

Security Center AI è una piattaforma di intelligence per report di sicurezza basata su Django (backend) e React/Vite (frontend). L'architettura segue il pattern MVC con separazione chiara tra:

  • Backend (Django): API REST, business logic, security layer
  • Frontend (React/Vite): UI, state management, client-side security

🔒 Backend - Security Hardening

1. Validazione SECRET_KEY Obbligatoria

security_center_ai/settings/base.py:96-99

# Validazione obbligatoria di SECRET_KEY if not SECRET_KEY: raise ValueError("SECRET_KEY environment variable is required")

📋 Dettagli Tecnici

La SECRET_KEY in Django è utilizzata per:

  • Criptografia delle sessioni
  • Generazione dei token CSRF
  • Firma dei cookie di sessione
  • Password reset tokens
  • Criptografia dei messaggi di sessione

⚠️ Rischi Senza Validazione

Senza questa validazione, il rischio include:

  • Deployment accidentale con chiavi di default (insecure)
  • Compromissione di tutte le sessioni utente
  • CSRF token prevedibili
  • Session hijacking facilitato

✅ Implementazione

La validazione avviene al caricamento delle settings, garantendo che l'applicazione non parta mai con una configurazione insicura. Questo è un fail-fast approach che previene problemi in produzione.

2. Security Headers Comprehensivi

security_center_ai/settings/base.py:184-197

# Security Headers Configuration SECURE_HSTS_SECONDS = 31536000 # 1 anno SECURE_HSTS_INCLUDE_SUBDOMAINS = True SECURE_HSTS_PRELOAD = True SECURE_SSL_REDIRECT = False # Configurabile SESSION_COOKIE_SECURE = False # Configurabile CSRF_COOKIE_SECURE = False # Configurabile SECURE_BROWSER_XSS_FILTER = True SECURE_CONTENT_TYPE_NOSNIFF = True X_FRAME_OPTIONS = 'DENY' SECURE_REFERRER_POLICY = 'strict-origin-when-cross-origin'

📋 Analisi Dettagliata degli Headers

Header Valore Scopo Protezione
Strict-Transport-Security max-age=31536000; includeSubDomains; preload Forza HTTPS Man-in-the-Middle attacks
X-Frame-Options DENY Previene framing Clickjacking
X-Content-Type-Options nosniff Previene MIME sniffing MIME-type confusion
X-XSS-Protection 1; mode=block Attiva filtro XSS Cross-site scripting
Referrer-Policy strict-origin-when-cross-origin Controlla referrer info Information leakage

🔒 HSTS (HTTP Strict Transport Security)

L'header HSTS con i parametri configurati garantisce:

  • max-age=31536000: Il browser ricorda di usare HTTPS per 1 anno
  • includeSubDomains: Tutti i subdomain devono usare HTTPS
  • preload: Il dominio può essere incluso nella HSTS preload list
⚠️ Nota Importante: HSTS preload è irreversibile per 1 anno. Assicurarsi che HTTPS sia configurato correttamente prima di abilitare questa opzione.

🛡️ X-Frame-Options: DENY

Previene completamente il framing della pagina in altri siti, proteggendo da:

  • Clickjacking attacks
  • UI redress attacks
  • Phishing via iframe

3. Validazione Input Robusta

security/views.py:104-122

📋 Pattern di Validazione Implementati

3.1 Validazione Severity (Whitelist)
❌ PRIMA - Vulnerabile
# Input diretto senza validazione if request.GET.get("severity"): alerts = alerts.filter( severity=request.GET["severity"] )
✅ DOPO - Sicuro
# Validazione contro whitelist severity = request.GET.get("severity") if severity and severity in [ s[0] for s in Severity.choices ]: alerts = alerts.filter(severity=severity)
3.2 Validazione Date
# Validazione rigorosa delle date date_str = request.GET.get("date") if date_str: try: from datetime import datetime date_obj = datetime.strptime( date_str, "%Y-%m-%d" ).date() alerts = alerts.filter( created_at__date=date_obj ) except ValueError: pass # Ignora formato invalido
3.3 Validazione Numeri
# Validazione ID numerici source_id = request.GET.get("source") if source_id and source_id.isdigit(): alerts = alerts.filter( source_id=int(source_id) )

🔒 Analisi dei Rischi Mitigati

Tipo di Input Rischio Senza Validazione Mitigazione
Severity SQL injection, query injection Whitelist di valori validi
Date Format injection, DoS try/except con formato specifico
Numeri Type confusion, injection isdigit() + type casting

✅ Best Practices Adottate

  • Whitelist over blacklist: Si accettano solo valori esplicitamente validi
  • Fail silently: Input invalidi vengono ignorati, non causano errori
  • Type safety: Validazione del tipo prima dell'uso
  • Format enforcement: Date devono seguire formato specifico

4. CSRF Protection Completa

security/views.py

📋 Implementazione

Il decorator @ensure_csrf_cookie è stato aggiunto a TUTTE le views API (15+ endpoint):

from django.views.decorators.csrf import ensure_csrf_cookie # Applicato a tutte le views @ensure_csrf_cookie def alerts_view(request): # ...

🔒 Come Funziona CSRF Protection

🔄 Flow CSRF:
  1. Client richiede una pagina → Server invia cookie CSRF
  2. Client invia richiesta POST → Include token CSRF
  3. Server verifica token → Confronta con cookie
  4. Se match → Request accettata
  5. Se no match → Request rifiutata (403)

⚠️ Perché è Importante

Senza CSRF protection, un attaccante può:

  • Eseguire azioni non autorizzate per conto dell'utente
  • Modificare configurazioni di sicurezza
  • Eliminare dati sensibili
  • Cambiare permessi utente

🎨 Frontend - Security & UX

1. Client CSRF Handler NUOVO

frontend/src/services/securityApiClient.csrf.ts

📋 Architettura del Handler

// Client-side CSRF Handler let csrfToken: string | null = null; // Recupero e caching del token async function getCsrfToken(): Promise { if (csrfToken) return csrfToken; const response = await fetch('/api/csrf/', { credentials: 'include' }); const data = await response.json(); csrfToken = data.csrfToken; return csrfToken; } // Interceptor per richieste non-safe async function secureRequest( url: string, options: RequestInit ): Promise { const method = options.method?.toUpperCase(); if (method && ['POST', 'PUT', 'DELETE'].includes(method)) { const token = await getCsrfToken(); options.headers = { ...options.headers, 'X-CSRFToken': token }; } return fetch(url, { ...options, credentials: 'include' }); }

🔒 Caratteristiche di Sicurezza

  • Automatic token retrieval: Il token viene recuperato automaticamente quando necessario
  • Caching: Il token viene cachato per evitare richieste ridondanti
  • Credentials include: I cookie vengono inviati con ogni richiesta
  • Error handling: Gestione robusta degli errori di rete
  • Type safety: TypeScript per type checking

🔄 Flow Completo

  1. Applicazione avviata → Nessun token in cache
  2. Prima richiesta non-safe → Recupero token da /api/csrf/
  3. Token cachato → Riutilizzo per richieste successive
  4. Token scaduto/invalidato → Nuovo recupero automatico

🛡️ Best Practices & Security Considerations

🔒 Security Principles Adottati

1. Defense in Depth

Multiple layers of security:

  • Validazione input (layer 1)
  • CSRF protection (layer 2)
  • Security headers (layer 3)
  • Authentication/authorization (layer 4)

2. Fail-Safe Defaults

Default configurations are secure:

  • SECRET_KEY required (fail-fast)
  • CSRF enabled by default
  • Secure cookies in production
  • HSTS enabled in production

3. Principle of Least Privilege

Minimal permissions required:

  • Permission checks on all endpoints
  • Role-based access control
  • Minimal database permissions

4. Secure by Design

Security built into architecture:

  • Input validation at boundaries
  • CSRF on all state-changing operations
  • Security headers by default
  • Secure session management
